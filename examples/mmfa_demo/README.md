# Deploying the Mobile Multi-Factor Authentication Cookbook

This demonstrations documents the required configuration to use IBM Verify Identity Access as a
Mobile Multi-Factor Authentication provider. This scenario configures a WebSEAL reverse proxy
to protect applications for mobile and browser based clients.

Users are able to register a mobile device as an authentication mechanism, and then use that mechanism
to authenticate or sign transaction data generated by IBM Verify Identity Access.


Kubernetes environment
----------------------

To deploy the required containers we will be using a kubernetes distribution called microk8s. However any Kubernetes or OpenShift environment will work.

We will be using the Verify Access Operator to manage the promotion of configurations to the runtime Reverse Proxy and Authorization containers.

Update the ingress addon to permit SSL passthrough. The following code will update the nginx ingress
controller provided with microk8s
```
kubectl patch daemonset -n ingress nginx-ingress-microk8s-controller --type='json' -p='[{"op": "replace", "path": "/spec/template/spec/containers/0/args", "value": [
  "/nginx-ingress-controller",
  "--configmap=$(POD_NAMESPACE)/nginx-load-balancer-microk8s-conf",
  "--tcp-services-configmap=$(POD_NAMESPACE)/nginx-ingress-tcp-microk8s-conf",
  "--udp-services-configmap=$(POD_NAMESPACE)/nginx-ingress-udp-microk8s-conf",
  "--ingress-class=public",
  "--publish-status-address=127.0.0.1",
  "--enable-ssl-passthrough"
]}]'
```


Installing the Verify Access Operator
-------------------------------------
The Verify Access Operator can be installed into any Kubernetes environment from source code

```
kubectl create -f https://github.com/IBM-Security/verify-access-operator/releases/download/v24.12.0/bundle.yaml
```

Creating PKI
------------
Use OpenSSL to generate the Public/Private RSA key pairs as well as X509 certificates.
You will need to do this for WebSEAL, LDAP, and PostgreSQL deployments.

It is important to note that many services now perform hostname verification with either the 
CommonName or the SubjectAlternateName entries of X509 certificates, so make sure the hostnames
being set are valid for you environment.

```
#Generate the WebSEAL Key/Certificate
openssl genrsa -passout pass:passw0rd -aes128 2048 > mmfa.key
openssl rsa -in mmfa.key -passin pass:passw0rd -out mmfa.key
openssl req -new -out mmfa.csr -key mmfa.key \
    -subj "/C=AU/O=IBM/OU=Security/CN=www.mmfa.ibm.com
cat > mmfa.cnf <<EOF
authorityKeyIdentifier=keyid,issuer
basicConstraints=CA:FALSE
keyUsage = digitalSignature, nonRepudiation, keyEncipherment, dataEncipherment, cRLSign, keyCertSign
subjectAltName = @alt_names

[alt_names]
DNS.1 = www.myidp.ibm.com
DNS.2 = myidp.ibm.com
DNS.3 = mmfa.myidp.ibm.com
DNS.4 = mobile.myidp.ibm.com


EOF
openssl x509 -req -signkey mmfa.key -in mmfa.csr -out mmfa.pem -days 9999 -extfile mmfa.cnf
rm mmfa.cnf mmfa.csr
openssl pkcs12 -export -out mmfa.p12 -inkey mmfa.key -name www.mmfa.ibm.com -in mmfa.pem --passout pass:Passw0rd

#Generate the PostgreSQL Key/Certificate
openssl req -x509 -newkey rsa:4096 -keyout postgres.key -out postgres.crt -days 3650 \
        -subj "/CN=postgresql/O=ibm/C=us" -nodes -addext "subjectAltName = DNS:ivia-mmfa-postgresql"
cat  "postgres.crt" "postgres.key" > "server.crt"

#Generate the OpenLDAP Keys/Certifcate
openssl dhparam -out "dhparam.pem" 2048
openssl req -x509 -newkey rsa:4096 -keyout ldap.key -out ldap.crt -days 3650 -subj "/CN=openldap/O=ibm/C=us" -nodes
cp ldap.crt ca.crt
```


Create Verify Access Containers
-------------------------------

Sample Kubernetes deployments have been provided in the `mmfa_demo_services.yaml` and `mmfa_demo_deployment.yaml` files.

Configuration Steps
-------------------
Run the MMFA configuration yaml to deploy this scenario.

We are going to use a Kubernetes Job to orchestrate the configuration tool; and we'll need to set the job 
up in a couple of steps.
You will need to:
 * Give the service account you are using to run the configuration tool permission to read secrets and 
    restart deployments within its own namespace
 * Deploy the Job (defined below) to the same namespace that you are deploying IVIA to
 * Copy the required configuration files to the newly created pod
   - note:: you could also use a pre-defined config map for this; however there are some limitations on 
            config maps (size limits of individual files) which are not compatible with some parts of 
            this demonstration (glowroot APM is typically larger the file size limit). Therefore we will
            use a Persistent Volume Claim instead, and copy the files to the mounted volume once the pod
            is running.
 * Tell the Job that all are present and configuration can begin. We will do this by creating an empty file
    that our Job will be watching for.

Kubernetes job to run all configuration
_______________________________________

```
apiVersion: batch/v1
kind: Job
metadata:
  name: mmfa-config
spec:
  template:
    spec:
      containers:
      - name: verify-identity-access-configurator
        image: autoconf:latest
        imagePullPolicy: Never
        volumeMounts:
        - name: mmfaconfigvol
          mountPath: /verify_access_config
        command:
        - "bash"
        - "-c"
        - |
          echo "Starting MMFA Config"
          IVIA_CONFIG_YAML=mmfa_config.yaml IVIA_MGMT_BASE_URL=https://isva-mmfa-config:9443 python3 -m ibmvia_autoconf;
        envFrom:
        - secretRef:
            name: mmfa-env
      restartPolicy: Never
      volumes:
      - name: mmfa-config
        configMap:
          name: mmfa-config
      - name: mmfaconfigvol
        emptyDir: {}
      initContainers:
      - name: config-volume-builder
        image: autoconf:latest
        imagePullPolicy: Never
        volumeMounts:
        - mountPath: /verify_access_config
          name: mmfaconfigvol
        - mountPath: /tmp/mmfa_config
          name: mmfa-config
        command:
        - "bash"
        - "-c"
        - |
          cp /tmp/mmfa_config/*.{p12,pem,yaml} /verify_access_config/
          unzip /tmp/mmfa_config/mapping_rules.zip -d /verify_access_config/
  backoffLimit: 2
```

Testing it out!
---------------

Once you have deployed this demonstration there are a few different authentication scenarios you can
use to test out the deployment.

Mobile demonstration appliction
_______________________________
The first scenario you are likely to want to test out is the username only login to view the mobile-demo 
application. This scenario lets a user authenticate with only ther username. This is not a strong level 
of authentication so you will not have sufficient permission to view all applications/features; however 
you will have sufficient permission to register MMFA/FIDO2 authenticators.

You will also be able to try out the transaction signing scenario.
